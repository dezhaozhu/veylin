# Compass 排产操作手册（agent 必读）

## 治理循环 —— 改排产的唯一正确姿势

任何对排产的修改都走**草稿通道**，绝不直接改：

```
propose_schedule_edit  →  preview_schedule_edit  →  commit_schedule_edit
        （提案）              （预览影响）           （确认后提交）
                                  ↘  discard_schedule_edits（放弃）
```

铁律：

1. **绝不跳过 preview。** 提案后必须先预览，把 before/after 的影响（延误变化、受影响订单数）如实呈现给用户。
2. **commit 之前必须停下来等用户确认。** 即使用户此前说过"帮我改"，具体到每一次提交，都要把预览结果给用户看并获得明确同意。禁止自动提交。
3. **诚实呈现。** 预览显示变差（延误增加、新增错期）时如实说，不粉饰；由用户决定是否仍然提交。
4. **可撤销 ≠ 可随意。** 提交有版本与回滚兜底，但这是安全网，不是许可证。
5. 改期类（due_at 等）编辑会进入**提案审批流**而非立即生效——要向用户说明"这将生成一个待审批提案"。

## 提交后的重排语义（增量提交）

commit 触发**增量重排**：只重解受影响的订单，其余排程冻结为背景（秒级返回）。这意味着：

- 提交结果通常几秒内可见；无需警告用户"会等很久"。
- 预览与提交使用同一套求解逻辑，preview 展示的就是 commit 会发生的。

## 只读分析优先

用户问"为什么/怎么样"类问题时，用只读工具回答（get_health、get_schedule_rows、鼓点/负荷分析），不要为了回答问题去发起编辑提案。

## 多厂 / project 边界

你手里的 token 绑定了一个 project（一个或多个厂的数据集）。**你无法也不应尝试切换数据集**——这是部署期由人配置的。如果用户要求访问 token 之外的厂，说明需要在 Compass 侧为其配置相应 project，并把这一点告诉用户。
